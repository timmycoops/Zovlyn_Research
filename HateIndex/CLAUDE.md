# Commodity Hate Index — Claude Code Brief

## What this is

A weekly-updating GitHub Pages site that ranks commodities by a composite **Hate Score** (capitulation signal) and overlays sector rotation (RRG) so the buy zone — hated AND rotating into Improving / Leading — is visible at a glance.

## Conventions (NON-NEGOTIABLE)

1. **Timezone**: all timestamps stored as ISO 8601 UTC strings. No naive datetimes anywhere.
2. **Frequency**: weekly bars (Friday close). No daily, no intraday.
3. **Storage**: parquet for raw + processed data, JSON for site output. Never CSV in the pipeline.
4. **Point-in-time correctness**: when computing the score for week T, use only data available at week T. CFTC reports lag — Friday COT release covers Tuesday's positioning, so lag CFTC data by 1 week minimum.
5. **No look-ahead**: forward-return columns and current-state columns must never coexist in the same DataFrame without explicit lag handling. The backtest is the only place forward returns are computed, and they're computed AFTER the signal is generated.
6. **Universe**: the 12 commodities in `docs/SPEC.md`. If you want to add or remove one, ask Tim first.
7. **Sources**: yfinance, CFTC.gov, FRED. Do NOT add scrapers without asking — they break and embarrass us.

## Working style

- **Ask before scraping.** If a data source needs HTML scraping (Yahoo, ETF.com, Stockanalysis), pause and ask Tim. Most have API alternatives that survive longer.
- **Pin versions.** Always update `requirements.txt` when adding a dependency. CI must be reproducible.
- **Logs over prints.** Use the `logging` module, not `print`. Set level via env var.
- **Type hints everywhere.** Even on private functions.
- **Small commits.** One logical change per commit. Conventional commit messages (`feat:`, `fix:`, `data:`, `docs:`).
- **Tests for math.** If a function computes a z-score, drawdown, or rotation status, it has a test.

## Don't

- **Don't rebuild the dashboard.** It's locked. If a feature requires changing it, raise with Tim.
- **Don't commit secrets.** FRED key, GDELT credentials → repo Settings → Secrets only.
- **Don't use daily bars.** Weekly only. The whole signal framework assumes weekly cadence.
- **Don't add a database.** Parquet + DuckDB read-on-demand is the design. No Postgres, no SQLite.
- **Don't write tests that hit the network.** Mock yfinance/CFTC responses with fixtures in `tests/fixtures/`.
- **Don't optimise prematurely.** A weekly refresh runs in <60s on the GitHub Actions runner. That's fine.
